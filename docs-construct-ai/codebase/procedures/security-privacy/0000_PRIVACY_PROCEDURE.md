## 1. EXECUTIVE SUMMARY

This document outlines the privacy procedures for Construct AI in compliance with data protection regulations in:

- **ZA (South Africa)**: Protection of Personal Information Act (POPIA) - Act 4 of 2013
- **GN (Guinea)**: Law L/2016/012/AN on the Protection of Personal Data
- **SA (Saudi Arabia)**: Personal Data Protection Law (PDPL)

This procedure ensures compliance with international data protection standards while maintaining operational efficiency in construction project management.

---

## 2. SCOPE AND APPLICABILITY

### 2.1 Geographic Coverage
- **ZA (South Africa)**: All operations involving South African citizens or data processing within South African jurisdiction
- **GN (Guinea)**: All operations involving Guinean citizens or data processing within Guinean jurisdiction
- **SA (Saudi Arabia)**: All operations involving Saudi citizens or data processing within Saudi jurisdiction

### 2.2 Data Subjects
This procedure applies to personal information of:
- Employees and contractors
- Project stakeholders (clients, suppliers, subcontractors)
- End users of the Construct AI platform
- Third-party service providers

### 2.3 Data Types Covered
- Personal identification information (`user_management`, `user_profiles`)
- Contact information (`contacts`, `user_emails`)
- Employment records (`personnel_records`, `timesheets`)
- Financial information (`financial_records`, `petty_cash`)
- Health and safety data (`safety_incidents`, `medical_records`)
- Project-related personal data (`travel_requests`, `training_materials`)

### 2.4 Exclusions
This procedure does not apply to:
- Anonymous data where individuals cannot be identified
- Publicly available information
- Data processed for national security purposes

## 3. LEGAL FRAMEWORK

### 3.1 ZA (South Africa) - POPIA
**Key Principles:**
- Accountability and processing limitation
- Purpose specification and further processing limitation
- Information quality and data subject participation
- Security safeguards and openness
- Cross-border information flows

**Responsible Party Obligations:**
- Designate Information Officer
- Maintain Processing Register
- Conduct Privacy Impact Assessments (PIAs)
- Implement security measures
- Handle data breach notifications within 72 hours

### 3.2 GN (Guinea) - Law L/2016/012/AN
**Key Principles:**
- Lawfulness, fairness, and transparency
- Purpose limitation and data minimization
- Accuracy and storage limitation
- Integrity, confidentiality, and accountability

**Data Controller Obligations:**
- Obtain explicit consent for sensitive data
- Implement appropriate security measures
- Maintain records of processing activities
- Notify authorities of data breaches within 72 hours
- Appoint data protection officer

### 3.3 SA (Saudi Arabia) - PDPL
**Key Principles:**
- Lawfulness, fairness, and transparency
- Purpose limitation and data minimization
- Consent and legitimate interests
- Accuracy and security

**Data Controller Obligations:**
- Obtain consent for processing personal data
- Maintain records of processing activities
- Implement security measures
- Notify NCA of data breaches within 72 hours
- Appoint data protection officer for large-scale processing

## 5. DATA PROCESSING PRINCIPLES

### 5.1 Lawful Basis for Processing
**ZA (POPIA):**
- Consent of data subject
- Contractual necessity
- Legitimate interests
- Legal obligation
- Public interest

**GN (Guinea):**
- Explicit consent
- Performance of contract
- Compliance with legal obligation
- Legitimate interests
- Vital interests

**SA (Saudi Arabia):**
- Consent
- Contract performance
- Legal obligation
- Legitimate interests
- Public interest

### 5.2 Data Subject Rights
**Common Rights Across Jurisdictions:**
- Right to be informed about processing
- Right of access to personal data
- Right to rectification of inaccurate data
- Right to erasure ("right to be forgotten")
- Right to restrict processing
- Right to data portability
- Right to object to processing

**Jurisdiction-Specific Rights:**
- **ZA:** Right to object to automated decision-making
- **GN:** Right to withdraw consent
- **SA:** Right to object to direct marketing

### 5.3 Data Retention
**Retention Schedule:**
- Employee records: 7 years after termination (ZA), 5 years (GN), indefinite (SA)
- Financial records: 7 years (ZA, GN), 10 years (SA)
- Training records: 5 years (ZA), 3 years (GN), indefinite (SA)
- Incident reports: 5 years (ZA), 3 years (GN), 10 years (SA)

## 7. DATA BREACH PROCEDURES

### 7.1 Breach Detection and Assessment
1. **Detection:** Security team monitors for breaches 24/7
2. **Assessment:** DPO evaluates breach severity within 24 hours
3. **Containment:** Immediate steps to contain breach
4. **Notification:** Notify affected individuals and authorities

### 7.2 Notification Timelines
- **ZA:** Notify Information Regulator within 72 hours
- **GN:** Notify Data Protection Authority within 72 hours
- **SA:** Notify NCA within 72 hours

### 7.3 Notification Content
- Description of breach
- Categories and number of affected individuals
- Contact details for further information
- Measures taken to mitigate harm
- Recommendations for affected individuals

## 9. PRIVACY IMPACT ASSESSMENTS (PIAs)

### 9.1 When PIAs Are Required
- High-risk processing activities
- New technologies or systems
- Large-scale processing
- Systematic monitoring
- Sensitive data processing

### 9.2 PIA Process
1. **Screening:** Determine if PIA required
2. **Scoping:** Define assessment boundaries
3. **Assessment:** Evaluate risks and mitigation
4. **Mitigation:** Implement controls
5. **Sign-off:** Obtain approval
6. **Review:** Annual reassessment

### 9.3 PIA Documentation
- Project description
- Data flows and processing activities
- Risk assessment
- Mitigation measures
- Residual risks
- Sign-off and approval

## 15. DATABASE-SPECIFIC IMPLEMENTATION

### 15.1 Table-Level Controls
Using the following database tables for privacy compliance:

#### User Data Tables
- `user_management`: Core user profiles - encrypt sensitive fields
- `user_profiles`: Extended profile data - implement access controls
- `user_emails`: Email communications - log access and retention
- `contacts`: Contact information - consent management required

#### Employment Records
- `personnel_records`: Employee data - restricted access, audit logs
- `timesheets`: Work records - retention per local laws
- `training_materials`: Training completion - 5-year retention

#### Financial Data
- `financial_records`: Payment data - PCI DSS compliance required
- `petty_cash`: Expense records - approval workflows

#### Safety and Health Data
- `safety_incidents`: Incident reports - sensitive data handling
- `medical_records`: Health information - strict consent requirements

#### Project Data
- `travel_requests`: Travel arrangements - location data privacy
- `contracts`: Contract data - legal compliance requirements

### 15.2 Row-Level Security (RLS)
Implement RLS policies based on:
- Organization membership (`organization_users`)
- Project assignments (`project_permissions`)
- Discipline access (`user_discipline_access`)
- Role-based access (`user_role_assignments`)

### 15.3 Audit Logging
- `audit_log`: All data access and modifications
- `security_audit_log`: Security-related events
- `document_access_logs`: Document access tracking
- `vetting_access_log`: Contractor vetting access
