# 0000 AI-Native Domain Prompt Generation Procedure

## Overview

This procedure defines the standardized process for transforming a discipline's domain knowledge into a comprehensive AI-native prompt that guides **OpenClaw coding agents to generate, modify, and validate code during development** (dev mode). It does NOT produce prompts for running applications in production (prod mode). The generated prompt is used when agents are in dev mode to write procurement components, workflows, APIs, UI elements, and other code artifacts.

It produces:

1. **Enhanced Domain Knowledge** with AI-native operations section (Part 6B/7B)
2. **Updated Glossary** with AI-native terminology
3. **AI-Native Prompt Template** for coding agent implementation (dev mode)
4. **Workflow Docs Remediation Plan** for updating existing workflow documentation

**Origin:** Applied successfully to Safety (HSE) 02400 and Civil Engineering 00850 disciplines. Both achieved comprehensive AI-native coverage with automation spectrum, document generation pipelines, multi-agent orchestration, and safety boundaries.

**Critical Distinction:** The prompts generated by this procedure are used by OpenClaw coding agents **during development** to generate code. They are NOT used by the running application in production. The automation spectrum and safety boundaries in these prompts define what code agents can generate independently versus what requires human architect review before merge.

## Scope

This procedure applies to all discipline domain knowledge files within the Construct AI memory system. It transforms traditional domain knowledge (role descriptions, technical competencies, behavioral patterns) into AI-native coding agent guidance that defines:

- What code agents can generate independently (full automation — no human review needed)
- What code agents can generate but requires human architect review before merge (augment)
- What code human architects must design with agent support (human-led)
- What code agents must NEVER generate (safety boundaries: authority bypasses, audit modification, etc.)

## Phase 1: Domain Knowledge Analysis & Classification

### Step 1.1: Read Existing Domain Knowledge
Read the full domain knowledge file to understand:
- What the discipline team/role does (not just what AI does)
- All task categories and responsibilities
- All document outputs produced by the discipline
- Regulatory and safety boundary considerations
- Existing AI integration (if any)

### Step 1.2: Classify Tasks by Data Type
For each task category in the discipline, classify inputs as:

| Classification | Definition | AI Pattern |
|---------------|------------|-----------|
| **Text-Native** | Specs, calculations, reports, codes, schedules, tables, emails, RFIs, meeting minutes | LLM-friendly: drafting, checking, template mapping |
| **Geometry-Native** | DWG, RVT, IFC files, drawings, BIM models, schematics, P&IDs | Requires extraction pipeline → structured JSON → LLM |
| **Structured Data** | Databases, CSV, API outputs, sensor readings | Direct processing, dashboard generation, analytics |
| **Human-Led Only** | Physical inspections, professional sign-off, regulatory submissions, disciplinary decisions | Human execution only, AI provides data/context |

**Output:** Task classification matrix documenting all discipline tasks and their data type.

### Step 1.3: Inventory Document Outputs
List all documents produced by the discipline team. For each document:

1. Document name
2. Trigger (what causes it to be produced)
3. Input data sources
4. Output format
5. Frequency
6. Human review requirement (none / review-only / approve-only / full human)
7. Regulatory requirement (yes/no, which regulation)

**Output:** Document inventory table mapping all discipline outputs.

### Gate 1 Checkpoint
- [ ] Task classification matrix complete
- [ ] Document inventory complete
- [ ] Data types classified for all tasks
- [ ] Lead Reviewer (PromptForge Sage) has reviewed classification

## Phase 2: AI-Native Domain Knowledge Enhancement

### Step 2.1: Add AI-Native Operations Section
Insert a new part after the existing content (typically "Part 6B" or "Part 7" depending on current structure). This section must include:

#### 2.1.1: Text-Native vs Geometry-Native Architecture
- Define text-native inputs for this discipline
- Define geometry-native inputs for this discipline
- Define the extraction pipeline architecture for geometry-native inputs
- Define guard rails: what LLM can and cannot do with geometry data

#### 2.1.2: Automation Spectrum
Create a comprehensive table of all discipline tasks categorized by automation level:

| Automation Level | Definition | Examples | Human Role |
|-----------------|------------|---------|-----------|
| **Full Automation** | AI executes end-to-end with final human review | Document generation from structured data, compliance checking, report drafting | Reviews and approves |
| **Augment AI + Human** | AI drafts/analyses, human validates and finalizes | Preliminary sizing, feasibility assessment, schedule analysis | Co-creates, validates, decides |
| **Human-Led, AI-Informed** | AI alerts or recommends, human decides | Stop-work decisions, risk acceptance | Decides |
| **Human-Led Only** | AI has no role | Final sign-off, expert testimony, regulatory submissions | Executes and decides |

#### 2.1.3: Document Generation Pipeline
Create four-phase document generation mapping:

| Phase | Document Types | AI Trigger | Output Format |
|-------|---------------|------------|--------------|
| **Phase 1: Pre-Construction/Pre-Design** | Plans, registers, assessments, ERPs | Data loaded | PDF, JSON |
| **Phase 2: Design** | Calculations, reports, specifications, BOQs | Analysis complete | PDF, Excel |
| **Phase 3: Operational/Construction** | Method statements, ITPs, RFIs, NCRs, reports | Event triggered | Structured forms |
| **Phase 4: Strategic** | Audit reports, compliance attestations, claims | Schedule/trigger | PDF, presentation |

Include pipeline architecture diagram and 6 template design principles:
1. Structured data injection, not raw LLM generation
2. Provenance tracking on all injected data
3. Conditional logic for optional sections
4. Regulatory accuracy and currency
5. Multi-language support where applicable
6. Human-readable with structured data export

#### 2.1.4: AI-Native Capabilities Beyond Automation
Document capabilities that go beyond simple automation:

| Capability Category | Discipline-Specific Examples |
|--------------------|---------------------------|
| **Predictive Intelligence** | [Discipline-specific predictions: risk hotspots, condition monitoring, trend analysis] |
| **Multi-Agent Orchestration** | [Discipline-specific workflow chain with agent handoffs] |
| **Computer Vision / IoT** | [Discipline-specific monitoring: defect detection, environmental sensing] |
| **Natural Language Interface** | [Discipline-specific queries: code queries, spec checking, precedent research] |
| **BIM / Digital Model Intelligence** | [Discipline-specific: clash detection, quantity extraction, change propagation] |

#### 2.1.5: AI Safety Boundaries
Define what must NOT be automated:

- **Non-Delegable Human Decisions:** List 5-7 decisions that must remain human
- **AI Must Always Disclose:** List 5 disclosure requirements when AI is uncertain

#### 2.1.6: Technical Architecture Recommendations
Table of 8-10 technical components with recommended approach (same pattern used for Safety and Civil):

| Component | Recommended Approach |
|-----------|---------------------|
| Document generation | Template engine with structured data injection |
| [Discipline-specific data] | Dedicated service; LLM works from structured data only |
| [Discipline-specific analysis] | Integration with specialized libraries |
| Computer vision | Dedicated CV models; not LLM for raw images |
| Knowledge retrieval | Vector database (RAG pattern) |
| Audit trail | Immutable log with provenance tags |

### Step 2.2: Update Version and Metadata
- Increment domain knowledge version by 1.0
- Verify all cross-references remain valid
- Update gigabrain_tags to include: "ai-native-operations", "document-generation", "automation-spectrum", "multi-agent-orchestration"

### Gate 2 Checkpoint
- [ ] Part 6B/7B section complete with all 6 sub-sections
- [ ] Automation spectrum table has 20+ tasks categorized
- [ ] Document generation pipeline has 15+ document types mapped
- [ ] Non-delegable decisions defined (5+)

- [ ] Technical architecture table has 8+ components
- [ ] Version incremented

## Phase 4: AI-Native Prompt Template Generation

### Step 4.1: Create New Prompt File
Create new file at:
```
docs-construct-ai/disciplines/[CODE]/agent-data/prompts/[CODE]_AI-NATIVE-[DISCIPLINE-NAME]-PROMPT.md
```

### Step 4.1.1: Clarify Development vs Production Purpose
The prompt must explicitly state in its Overview section that:
- This prompt is for **OpenClaw coding agents operating in dev mode**
- Agents use this prompt to **generate, modify, and validate code**
- This prompt is NOT for the running application in production
- The automation spectrum defines what **code** agents can generate independently vs. what requires human architect review

### Step 4.2: Structure Content
The prompt must include these sections (in order):

1. **Metadata Header**
   - title, description, version: 1.0, memory_layer, para_section, gigabrain_tags (include all relevant tags), openstinger_context, last_updated, related_docs

2. **Overview** — What this prompt does, how it connects to domain knowledge, key lesson from previous disciplines

3. **Implementation Action List & Progress Tracking** — 8-phase checklist:
   - Phase 1: Foundation (Text-Native vs Geometry-Native Architecture)
   - Phase 2: Document Generation Pipeline
   - Phase 3: Multi-Agent Orchestration
   - Phase 4: Computer Vision & Site Monitoring
   - Phase 5: Natural Language Interface
   - Phase 6: [Discipline-specific intelligence: BIM, predictive, etc.]
   - Phase 7: Data-Driven [Discipline]
   - Phase 8: AI Safety Boundaries & Governance

4. **Discipline Context** — Scope summary, document types, related disciplines

5. **Core Template Structure** — PARA Navigation + Gigabrain Search + Memory Context + Discipline-specific AI-Native Context + Knowledge Integration

6. **Discipline-Specific Use Case Templates** (4 minimum):
   - Use Case 1: Data Processing Pipeline (DWG/BIM/input extraction)
   - Use Case 2: Document Generation Pipeline Execution
   - Use Case 3: Intelligence Implementation (BIM/predictive/CV)
   - Use Case 4: Multi-Agent Orchestration

   Each use case must include: PARA Navigation, Gigabrain Search, Memory Context, Discipline AI-NATIVE CONTEXT, Required output structure.

7. **Automation Spectrum** — Same table as domain knowledge, repeated for prompt context

8. **Document Generation Pipeline** — Pipeline diagram and 6 template design principles

9. **AI-Native Capabilities Beyond Automation** — All capability categories with discipline-specific examples

10. **AI Safety Boundaries** — Non-delegable decisions + AI must always disclose

11. **Technical Architecture Recommendations** — Same table as domain knowledge

12. **Agent Coordination Workflow** — Mermaid or text diagram of agent orchestration

13. **Implementation Best Practices** — 6+ coordination guidelines + 6+ boundary enforcement rules

14. **Success Metrics** — Document generation, data processing, intelligence, multi-agent metrics with specific targets

15. **Version History** — What was included in v1.0

16. **Behavioral Rules** — 10+ non-negotiable rules for coding agents (what code they can/cannot generate)

### Step 4.3: Cross-Reference Verification
After creation, verify:
- [ ] All related_docs paths are valid
- [ ] All gigabrain_tags match actual content
- [ ] All use case templates reference valid domain knowledge sections
- [ ] Automation spectrum matches domain knowledge exactly
- [ ] Technical architecture recommendations match domain knowledge

### Gate 4 Checkpoint
- [ ] Prompt file created with all 16 sections
- [ ] 4+ use case templates complete
- [ ] Automation spectrum matches domain knowledge
- [ ] Document generation pipeline matches domain knowledge
- [ ] 10 behavioral rules defined
- [ ] Success metrics defined for all AI capability categories
- [ ] Cross-references verified
