# 02050 Information Technology Domain Knowledge

## Part 1: Discipline Overview

### 1.1 Role and Purpose
The 02050 Information Technology discipline encompasses all technology infrastructure, system architecture, software development, error management, and AI integration within the ConstructAI platform. IT operations serve as the backbone for all other disciplines, providing:

- System architecture and infrastructure management
- Error tracking and analytics platform
- AI/ML integration and prompt management
- Team collaboration tools and workflows
- Template and form management systems
- Chatbot and conversational AI systems
- Database management and schema design
- Security and access control systems
- Document processing and OCR capabilities
- External API management and integration

## Part 4: Regulatory and Compliance Requirements

### 4.1 Data Protection Standards
- **GDPR/Data Privacy**: All user data processing must comply with applicable privacy regulations
- **Data Retention**: Define retention policies for error logs, chat histories, and system data
- **Access Control**: Implement role-based access with RLS policies for all sensitive data
- **Audit Logging**: Maintain immutable logs for all system changes and data access

### 4.2 Security Standards
- **Authentication**: Secure authentication for all user and API access
- **Encryption**: Encrypt sensitive data at rest and in transit
- **Vulnerability Management**: Regular security scanning and patch management
- **Incident Response**: Documented procedures for security incident handling

### 4.3 Quality Standards
- **Code Review**: All code changes require peer review before deployment
- **Testing Standards**: Unit tests, integration tests, and E2E tests for critical paths
- **Documentation Standards**: All system changes must be documented
- **Change Management**: Formal change control for production deployments

## Part 7: AI-Native Operations

### Part 7A: Text-Native vs Geometry-Native Classification

For the Information Technology discipline, inputs are classified as:

| Classification | Description | Examples | AI Pattern |
|---------------|-------------|----------|-----------|
| **Text-Native** | Code files, documentation, error logs, configuration files, API responses, user queries | LLM-friendly: parsing, analysis, generation |
| **Structured Data** | Database records, JSON configs, API payloads, metrics, logs | Direct processing, dashboard generation, analytics |
| **Code-Native** | JavaScript, React components, SQL schemas, TypeScript types | AST parsing, static analysis, code generation |
| **Visual/Binary** | Screenshots, UI mockups, architecture diagrams | Requires image analysis pipeline |
| **Human-Led Only** | Architectural decisions, security sign-offs, production deployments | Human execution, AI provides data/context |

### Part 7B: Automation Spectrum

| Automation Level | Definition | Examples | Human Role |
|-----------------|------------|---------|-----------|
| **Full Automation** | AI executes end-to-end with final human review | Error classification, log analysis, template generation, basic code documentation, configuration validation, dashboard data population | Reviews and approves |
| **Augment AI + Human** | AI drafts/analyses, human validates and finalizes | Code generation from templates, error pattern analysis with recommendations, documentation drafting, API integration scaffolding, test case generation | Co-creates, validates, decides |
| **Human-Led, AI-Informed** | AI alerts or recommends, human decides | Security incident response, production deployment decisions, system architecture changes, critical bug triage | Decides |
| **Human-Led Only** | AI has no role | Production database migrations, security policy changes, vendor negotiations, team restructuring | Executes and decides |

### Part 7C: Document Generation Pipeline

| Phase | Document Types | AI Trigger | Output Format |
|-------|---------------|------------|--------------|
| **Phase 1: System Documentation** | Architecture diagrams, schema docs, API guides, security reports | System change detected | MD, PDF, JSON, ERD |
| **Phase 2: Error Management** | Error reports, resolution workflows, health reports, analytics | Error event or scheduled | PDF, MD, Web UI |
| **Phase 3: AI Operations** | LLM configs, prompt libraries, AI performance reports, voice call guides | AI system change or schedule | MD, JSON, PDF |
| **Phase 4: Operational** | Meeting minutes, method statements, risk assessments, knowledge base articles | Meeting completion, risk event | MD, PDF, Web UI |
| **Phase 5: Strategic** | IT roadmap, capability assessments, compliance audits, vendor reports | Quarterly review | PDF, Presentation |

**Template Design Principles:**
1. Structured data injection, not raw LLM generation
2. Provenance tracking on all injected data
3. Conditional logic for optional sections
4. System accuracy and currency
5. Multi-environment support (dev, staging, prod)
6. Machine-readable with human-friendly export

### Part 7E: Technical Architecture Recommendations

| Component | Recommended Approach |
|-----------|---------------------|
| Code generation | Template-based generation with AST validation |
| Error tracking | Structured logging with vector search for pattern matching |
| Document generation | Template engine with structured data injection from APIs, databases |
| Log analysis | Dedicated log aggregation; LLM works from structured summaries |
| Code analysis | Static analysis tools (ESLint, TypeScript compiler); LLM for guidance only |
| Knowledge retrieval | Vector database (RAG pattern) for docs, configs, error solutions |
| Audit trail | Immutable log with provenance tags for all AI-generated outputs |
| AI orchestration | Agent coordination with structured data handoffs |

### Part 7F: AI Safety Boundaries

**Non-Delegable Human Decisions:**
1. Production deployment approvals for database migrations
2. Security policy changes and RLS policy modifications
3. System architecture decisions affecting multiple disciplines
4. Critical bug triage and priority decisions for customer-impacting issues
5. Vendor selection and contract decisions
6. Data retention policy changes
7. Encryption key management and rotation decisions

**AI Must Always Disclose:**
1. When system state data is stale or potentially inaccurate
2. When error analysis is based on incomplete log data
3. When code generation recommendations have not been tested
4. When security analysis has limited coverage or false positive risk
5. What manual verification steps remain before production deployment
6. When AI confidence in error classification is below threshold
7. What system components may be affected by proposed changes
